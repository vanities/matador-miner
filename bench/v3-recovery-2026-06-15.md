# v3 throughput-recovery config sweep (2026-06-15)

Formal benchmark of the v3 solo-mining **config levers** identified by the 2026-06-15
deep dive (three parallel agents: CUDA solve-path, overlap/bench tuning, 5090
ceiling). Goal: find config-accessible throughput to claw back toward "full V2
levels." Harness: `bench/v3-config-sweep.sh`.

## Method
- Image `btx-miner:local` (0.32.11 + our v3 pipeline-overlap), RTX 5090, stock 575W (no undervolt).
- `btx-matmul-solve-bench` in **v3 mode**: `--n 512 --b 16 --r 8 --backend cuda --nonce-seed-height 0 --product-digest-height 0 --parent-mtp-seed-height 130500 --parent-mtp <live mediantime> --block-height <tip>`.
- Sweep one knob at a time from the current live winner: `async=1 prefetch=3 workers=4 gpu-inputs=1 batch=2048`.
- **Idle-gate-paused** (btxd stays up; no warmup, no lost social credit; auto-resume on exit). btxd was never restarted.
- median nonce/s from the bench JSON; watts = active-power mean (>150W) over the run.

## Verdict: NO config lever recovers throughput

Every knob lands within **-10% .. +1%** of the current config. batch is flat,
prefetch slightly hurts, more prepare-workers actively hurt (CPU over-subscription).
**The current config (`async=1 prefetch=3 workers=4 gpu-inputs=1 batch=2048`) is
already optimal** for this bench. Matches the v2-honest sweep's conclusion: the
pipeline is GPU-bound, not config-bound.

### Warm-up artifact (methodology lesson)
A first 8-iter "smoke" run showed `workers=8 = 1.58x` and `batch=8192 = 1.06x`.
That was **false** - a GPU clock-ramp artifact. The 8-iter baseline ran cold
(117k n/s / 155W, idle clocks straight out of the idle-gate pause) so every later
run looked faster. At 60 iters everything runs warm and the gains vanish. **A
one-shot bench lies; always warm + use enough iters.** (TODO for the harness: a
discarded warm-up run before the baseline.)

## What this means for recovering throughput
- **Config tuning is exhausted** - no free wins in batch/prefetch/workers.
- The hot **kernels are already tapped out**: all six PR#58 optimizations
  (windowed-SHA scanner + matrix-gen midstate, factored 2x2 matmul, fused single
  reduction) are upstreamed into 0.32.11 and exercised at v3 mainnet.
- The v3 penalty vs v2 is **mostly consensus-fixed**: v3 folds the nonce into the
  matrix seed (anti-ASIC), so the base matrix can no longer be amortized per-tip
  like v2. That's the bulk of "~50-66% of V2" and it is by design.
- **Caveat - the bench is digest-bound:** the dominant LIVE rate (~14-15k
  digest-attempts/s) is also digest-bound, so the batch lever (which acts on the
  pre-hash SCAN path) is unlikely to move the live block-finding rate either; a
  live `BTX_MATMUL_SOLVE_BATCH_SIZE` test would cost a miner restart (warmup +
  social-credit reset) for a result the bench predicts is ~flat - not worth it.

### Remaining (hard) levers - NOT config
- Scheduling/scan code patches: double-buffer the scan to hide its blocking
  `cudaStreamSynchronize`; GPU-side survivor compaction; scan-occupancy via
  `epsilon`. Each needs implementation + a byte-exact AB gate, and only helps if
  the scan (not the digest) is the live limiter - which it is not at current difficulty.
- New kernel math (tensor-core M31 GEMM) - large R&D, parked; the matmul is not
  the bottleneck (matrix-gen SHA is, and it's already midstate-optimized).
- Protocol-level: shib's "back to V2" deep dive (consensus/seed-derivation) - the
  real lever, outside our solo client.

**Bottom line: we are at the config + kernel ceiling for the v3 solo client. Further
throughput recovery is consensus/protocol work (shib) or deep kernel R&D, not knobs.**
Solo + our overlap (+22.9%) remains the best client-side throughput we can run.

## Scan-path follow-up (epsilon × batch sweep)

Question: is the pre-hash SCAN the recoverable bottleneck (Agent 1's hypothesis)?
Swept `--epsilon-bits` (gate strictness: low eps = scan-dominated, high eps =
digest-dominated) × batch, v3 mode, idle-gate-paused.

| epsilon | batch=2048 n/s | batch=32768 n/s |
|---|---|---|
| 8 (scan-dominated) | 74,730,891 | 74,057,371 |
| 12 | 6,916,011 | 7,222,462 |
| 16 | 647,750 | 632,327 |
| 20 | 47,572 | 46,286 |
| 24 (digest-dominated) | 18,059 | 18,089 |

**Findings:**
- The **scan runs at ~74M nonces/s** (scan-dominated end) - ~5,000x the live digest
  rate (~14k/s). **The scan is NOT the live bottleneck**; the digest (matrix-gen SHA)
  is. Optimizing the scan sync/D2H/CPU-loop (double-buffer, GPU compaction) would
  speed up a non-bottleneck -> ~0 live gain. **Scan-path patches: not worth it.**
- **batch is flat (+-4%) in EVERY regime** - definitively not a lever, scan- or digest-bound.

## FINAL verdict (both formal benches)
Throughput at v3 is **digest-bound (matrix-gen SHA), kernel-maxed (PR#58 upstreamed),
and consensus-fixed (per-nonce matrix regen, anti-ASIC).** Neither config tuning nor
scan-path code changes recover it. The only client-side fruit left is marginal
(seed-midstate w[16..22] ~+2%, or tensor-core M31 GEMM = big R&D on the *matmul*,
which isn't even the bottleneck). **Real recovery is shib's protocol/seed-derivation
work.** Solo + our +22.9% overlap is the client-side ceiling.

## Scan-path CUDA patch designs (SHELVED — for reference only)

A patch-design pass produced two concrete scan-path patches; both are SHELVED
because the bench above proved the scan isn't the bottleneck. Code-structure
analysis independently agreed: scan orchestration is single-digit % of per-window
wall-clock, and the overlap patch already hides it behind the digest.

1. **Double-buffered scan** (hide the blocking `cudaStreamSynchronize`): two flag
   buffers + a `cudaEvent_t` so scan(N+1) launches before scan(N) is reaped. Est.
   gain **0-2%** (the sync is already hidden in overlap mode). Risk: medium.
2. **GPU-side survivor compaction** (atomic-append passing indices; copy back only
   ~count, not the full flag array; host-sort to restore ascending-nonce order for
   byte-exactness): removes the full-flag D2H (~2^18x fewer bytes) + the CPU loop.
   Est. gain **1-3%**. Risk: medium-high (atomic ordering = silent consensus-divergence
   if the sort is missed; touches the scan kernel + needs a Metal mirror).

Both need a byte-exact AB gate (reuse `patches-overlap/btx-matmul-overlap-ab.cpp`).
Key anchors (btx 0.32.11 @ 215170f2): scan kernel `src/cuda/oracle_accel.cu:976-1046`;
host driver/sync `oracle_accel.cu:1343-1439` (sync :1424, D2H :1417-1422, CPU loop
:1431-1435); window builder + 2nd survivor loop `src/pow.cpp:1499-1648`; Metal mirror
`src/metal/oracle_accel.mm:1091-1209`.

**Revisit ONLY if the bottleneck shifts** (e.g. difficulty drops making the scan
relevant, or a protocol change rebalances scan-vs-digest). Today: not worth touching
consensus-adjacent CUDA for <=3%.
