# Metal backend on M4 Max (MacBook Pro) - 2026-06-15

First pass at running + optimizing our miner on Apple Metal (this MacBook Pro,
M4 Max, 40-core GPU, Metal 4). Goal: optimize the Metal matmul-PoW kernels;
miner eventually pointed at the pc's btxd for GBT work.

## TL;DR

- btx ships a REAL, implemented Metal backend (`src/metal/`: shaders + `.mm`
  glue + tests + `btx-matmul-metal-bench`), default-ON for Apple builds. This is
  NOT a from-scratch CUDA->Metal port; it builds and runs natively today.
- Native macOS build works (no Docker - Docker on Apple Silicon can't reach the
  GPU; no CUDA on this machine).
- **The matmul kernel is MEMORY-BANDWIDTH-BOUND on this GPU, not ALU-bound.**
  This is the pivotal finding and it redirects the whole optimization strategy.
- Our CUDA "+22.9% overlap" patch does NOT apply to Metal (different pipeline).
  "Optimized on metal" = optimizing the Metal kernels specifically.

## Native build recipe (reproducible)

Requires full Xcode + the Metal Toolchain component:

```bash
xcodebuild -downloadComponent MetalToolchain   # one-time; `metal` CLI compiler
git clone --filter=blob:none https://github.com/btxchain/btx.git
cd btx && git checkout 215170f27f7d6889ce34aa7dbba2858ea07a468c   # v0.32.11
cmake -B build-metal -S . -DCMAKE_BUILD_TYPE=Release -DBTX_ENABLE_METAL=ON \
  -DBUILD_UTIL=ON -DBUILD_DAEMON=OFF -DBUILD_CLI=OFF -DBUILD_GUI=OFF \
  -DBUILD_TESTS=OFF -DBUILD_BENCH=OFF -DBUILD_TX=OFF -DENABLE_WALLET=OFF
cmake --build build-metal --target btx-matmul-metal-bench --parallel $(sysctl -n hw.ncpu)
./build-metal/bin/btx-matmul-metal-bench --batch-size 1 --iterations 25 --warmup 6
```

## Baseline (n=512, b=16, r=8, batch=1)

- Warm steady state: **~1400 us/digest (~700 digest/s)**. `submit_wait_us`
  (GPU compute) is ~95% of `per_digest_us`; CPU encode is negligible (~5 us).
- Cold first run ~2700 us; the GPU clocks up after a few iterations. Laptop
  thermals add real run-to-run variance (saw 1400 -> 4700 under load) - measure
  interleaved A/B, warm, and ideally plugged in / cool.

### Knob sweep (BTX_MATMUL_METAL_*)

- `PIPELINE=fused` and any `batch-size >= 2` are PATHOLOGICAL: ~28000 us/digest
  (~17x slower). `auto` correctly avoids the fused path at batch=1. The batched
  path is effectively broken for throughput - treat batch=1 as the only sane mode
  until that path is fixed.
- `FUNCTION_CONSTANTS=on` and `GPU_INPUTS=1`: each ~within noise on throughput,
  but the combo is *tighter* (1426-1452 vs auto's 1336-1548). Marginal.
- Net: env knobs are not where the win is. The kernel compute is.

## Why memory-bound (the diagnosis)

1. Modulus is `0x7fffffff` = **M31 = 2^31-1** (Mersenne prime). Replaced the hot
   `mul_mod`/`dot_step` (which use 64-bit `ulong` - emulated/slow on Apple's
   32-bit-native GPU) with a 32-bit `mulhi`-based M31 fold (2^32 == 2 mod M31).
   - Proven **byte-exact** vs the stock `reduce64` over 50M random pairs + field
     boundaries (residue is identical; safe for consensus).
   - Result: **perf-neutral** (~1400 us both, within noise). No win.
2. Arithmetic-intensity probe: inflated the inner-loop multiply ~15x with
   identical memory traffic -> **runtime did NOT increase**. Decisive: the matmul
   is bound by global-memory bandwidth, not ALU.

Both experiments reverted; kernel is back to stock. The 32-bit M31 form is worth
keeping in the back pocket - it becomes relevant only AFTER memory traffic is cut
(when ALU starts to matter again).

## The real fix (next session): tile `fused_prefix_compress`

`fused_prefix_compress` (the default hot kernel, kernels file ~line 845) has NO
tiling: each of the 256 threads in a b*b output tile independently reads its full
`a_prime` row (contiguous) and a **strided** `b_prime` column from global memory.
A tile reads `256 * 2n` elements but only needs `2*b*n` distinct values ->
~**b = 16x redundant global traffic**, which is exactly the bottleneck.

`build_prefix_tiled` (kernels file ~line 801) already demonstrates the fix:
stage 16x16 tiles of `a_prime`/`b_prime` into `threadgroup` memory per `ell`,
barrier, then multiply from threadgroup memory. Plan:

1. Write `fused_prefix_compress_tiled` = build_prefix_tiled's threadgroup staging
   fused with the existing weight + threadgroup reduction (instead of writing
   `c_prefix`). Also do a specialized (function-constant) variant since `b` is a
   compile-time constant there (static threadgroup arrays + full unroll).
2. Wire pipeline load + an env-gated dispatch (e.g. `BTX_MATMUL_METAL_TILED=1`)
   so it A/Bs cleanly against stock through all paths (legacy/fused/specialized,
   transcript/product modes; solve fns at matmul_accel.mm ~3060/3460/3870).
3. Gate on **byte-exact** end-to-end digest equality vs stock before keeping it.
4. Re-measure warm + interleaved on a cool machine.

Secondary ideas (lower priority, ALU-side - only matter post-tiling):
- Lazy reduction: fold every K products instead of every product (the NV miner's
  `d_dot` does this). Metal's `dot_step` folds every step.
- 2x2 register blocking (NV miner does `acc00/01/10/11`) - more output reuse per
  loaded value, compounds with tiling.

## CORRECTION (later same day): I was optimizing a DEAD kernel

Everything above about tiling/simd-reducing `fused_prefix_compress` is moot: a
GPU trace + a sabotage test proved that kernel is **never dispatched** for the
transcript bench. Hard-won lessons:

- **GPU trace method** (zero code change): `xcrun xctrace record --template
  "Metal System Trace" --output m.trace --launch -- <bench>`, then `xctrace
  export --xpath '.../table[@schema="metal-gpu-intervals"]'`. Parser:
  `bench/../parse-gpu-trace.py` (kept in the btx clone). Shows GPU time by
  channel + the btx compute-dispatch duration histogram.
- **Trace result:** per digest = ONE ~1900 us compute dispatch (~81%) + ONE
  ~429 us dispatch (~18%); SHA/matrix-gen negligible.
- **Sabotage test (critical):** before trusting ANY kernel A/B, make the kernel
  emit deliberately wrong output and confirm the digest changes. Mine didn't ->
  the gated kernel wasn't live. All three "washes" (M31, tiling, simd-reduce)
  were stock-vs-stock. ALWAYS sabotage-verify the kernel is on the live path
  first.
- **The live path:** the batch=1 transcript bench takes the LEGACY pipeline
  (`use_legacy_pipeline=1` via `ShouldUseLegacyTranscriptPipeline`), with
  function-constant specialization ON. Live kernels = `build_perturbed_specialized`
  -> `build_prefix_specialized` -> `compress_prefix_specialized`. NOT the fused
  kernel.
- **Real hot kernel = `build_prefix_specialized`** (kernels file ~line 607):
  one thread per output element, NO threadgroup / NO barriers / NO reduction.
  It writes the FULL prefix tensor `c_prefix` (N*n^2 = 32*512^2*4 = **32 MB per
  dispatch**) to global memory, which `compress_prefix_specialized` then reads
  back. The legacy path's signature cost is materializing + round-tripping that
  32 MB tensor - exactly what the fused path avoids, yet `auto` picks legacy.
- The ALU probe DID cover this kernel (dot_step is shared) -> still not
  ALU-throughput-bound. But tiling/barrier experiments were on the dead kernel,
  so those negatives say nothing about build_prefix.

### Corrected next steps (sabotage-verify each)
1. Re-probe `build_prefix_specialized` LIVE: does cutting the c_prefix write
   (e.g. write only ell==N-1) collapse the ~1900 us? That isolates write-traffic
   vs compute-latency. NOTE 32 MB / 1900 us = ~17 GB/s, far below the M4 Max's
   ~400 GB/s, so raw write bandwidth is likely NOT it -> suspect memory LATENCY
   / occupancy or the strided b_prime column reads.
2. Why does `auto` pick legacy over the (round-trip-free) specialized fused path?
   If the specialized fused kernel is correct + faster, getting it selected for
   transcript mode could remove the 32 MB round-trip wholesale. Generic fused is
   pathological (28000 us); specialized fused is unmeasured.
3. Only after a LIVE, sabotage-verified bottleneck: register-block build_prefix
   (2x2, NV-miner style) and/or reduce b_prime read latency.

## THE ACTUAL WIN: concurrency, not kernel micro-opt (~5.5-6.5x, byte-exact)

Knock-out probes on the LIVE `build_prefix_specialized` (each sabotage-verified -
digest changed every time):

| probe (32-thread shape, batch=1)        | per_digest_us | verdict |
|-----------------------------------------|---------------|---------|
| stock                                   | ~1388         | -       |
| write-knockout (skip 31/32 c_prefix writes) | ~1400     | NOT write-bound |
| read-knockout (no global loads in inner loop) | ~1350   | NOT read-bound |
| iteration-knockout (ell 32 -> 4, 8x less work) | ~1255  | build_prefix is only ~11% of the digest! |
| 15x ALU probe (earlier)                 | flat          | NOT ALU-bound |

Nothing inside the kernel moved the time -> the cost is NOT compute. The trace's
"build_prefix = 81%" was a cold + profiling-overhead artifact. The real limiter
is **submit/sync latency + GPU under-occupancy at one in-flight digest** (each
kernel is its own command buffer with a CPU<->GPU round-trip; the GPU idles
between them).

Proof = parallel sweep (aggregate throughput, the number that matters for mining):

| --parallel | agg digest/s | vs p=1 |
|-----------:|-------------:|-------:|
| 1          | 603          | 1.0x   |
| 2          | 1069         | 1.8x   |
| 4          | 1802         | 3.0x   |
| 8          | 2030         | 3.4x   |
| 16         | 3327         | 5.5x   |
| 32         | 3879         | 6.4x   |
| 48         | 4116         | 6.8x   |
| 64         | 4214         | 7.0x   |

Plateau ~4200/s; practical knee at **parallel 16-32 (~3300-3900/s)**. Pool-slots
/ async knobs add little once parallelism is up. `--parallel N` = N std::threads
each running the full solve on the shared Metal context (bench line 542) =
exactly what the real miner does via **`BTX_MATMUL_SOLVER_THREADS`** (read at
matmul_accel.mm:1261).

### ACTIONABLE: run the Metal solver with ~16-32 concurrent in-flight digests.
- `BTX_MATMUL_SOLVER_THREADS=16..32` (+ `BTX_MATMUL_METAL_POOL_SLOTS` >= that).
- This is byte-exact (just more concurrent solves) - NO kernel/consensus change,
  NO risk. ~5.5-6.5x more nonces/sec on the M4 Max than a serial solve.
- Answers the earlier "why aren't my CPU cores busy" question: at p=1 they
  aren't (one thread blocks on the GPU). At p=16-32 you'll see many cores busy
  feeding the GPU, and the GPU saturates.

Kernel micro-opt (M31, tiling, ILP) is a DEAD END until the solver is already
running concurrent - the kernel was never the bottleneck.

## LIVE POOL TEST: the "6x concurrency" win does NOT transfer to real mining (2026-06-15, later)

Tested the bench's headline concurrency win LIVE via matador-miner `--mode pool`
against shib's pool (minebtx.com:3333), Metal backend, on this M4 Max. Measured
by the miner's own `nonce/s` heartbeat (the real-mining number, not the synthetic
bench). All runs: 0 rejects, clean stratum handshake, jobs at height ~132057-132061.

| config | nonce/s | note |
|---|---:|---|
| baseline: 1 host thread, default batch | ~1210 | batch=192, digest-batch/s=6 |
| host-thread fan-out, 24 disjoint nonce lanes | ~1180 | FLAT (impl'd in RunPoolLoop, then reverted) |
| 1 host thread + `BTX_MATMUL_SOLVER_THREADS=16` + `METAL_POOL_SLOTS=16` | ~1130 | FLAT (backend pool-slots, the "right" lever) |
| `BTX_MATMUL_SOLVE_BATCH_SIZE=1` + slots=16 (the bench's 6x config) | **715** | WORSE |

### Why the bench's 6x is a strawman for live mining
- The 603->3879/s (6.4x) parallel sweep was the **batch=1 matmul-ONLY** path:
  one digest per GPU submission, heavily submit/sync-latency bound, so overlapping
  many in-flight digests hides the bubble.
- The **live pool path is the batch=192 nonce-PREFILTER path**: GPU input-gen +
  epsilon prefilter produces 192 candidate nonces per round and only the ~10%
  survivors get a full matmul digest. That batch already keeps the GPU busy ->
  it is GPU-throughput-bound and **concurrency-immune**. Adding host threads or
  pool slots does nothing; dropping to batch=1 to *enable* concurrency throws away
  the cheap prefilter and is net-NEGATIVE (715 < 1130).
- On Metal, `ResolveSolveBatchSize` returns 1 or 2 for the matmul digest; the
  "batch=192" in the stats is the nonce batch, a different thing. The pool slots
  (`ResolveMetalAutoPoolSlotCount`, default ~5 on a high-perf host) gate matmul
  concurrency that the prefilter path never needs.

### Verdict
REJECTED. No concurrency win is available in live Metal pool mining. The
RunPoolLoop fan-out was reverted (tombstone comment left in matador-miner.cpp).
This is the same bench-strawman trap as the CUDA btx-matmul-metal-bench batch=1
path - always A/B against the live `nonce/s` heartbeat, never the synthetic bench.

## NONCE-SEED BATCH SIZE sweep: 192 default is already optimal (2026-06-15, iter 2)

Hypothesis: the nonce-seed batch (pow.cpp:1291 `ResolveMetalNonceSeedBatchSize`)
is set to 192 by a COARSE tier (`gpu_core_count >= 30 -> 192`, next tier `>=60 ->
256`). The M4 Max has 40 cores - mid-bucket - so 192 might underfill a 40-core GPU;
a larger batch could raise throughput. Tunable live via `BTX_MATMUL_NONCE_SEED_BATCH_SIZE`
(clamped 1-4096), byte-exact (same per-nonce digests, just more candidates/round).
Tested live vs shib's pool (minebtx.com:3333), Metal, single stream, 0 rejects.

Coarse sweep (thermal drift confounded - bracket 192 fell 1200->1022 over 5 runs):
192=1200, 256=1202, 384=1131, 512=1083, 192(again)=1022 nonce/s.

Drift-controlled interleaved A/B (192 vs 256, alternating, warm regime):
- batch=256: 1049, 1093, 1076 -> avg 1073
- batch=192: 1105, 1099, 1060 -> avg 1088

VERDICT: REJECTED. 192 marginally beats 256 (~1.4%, within noise); 384/512 clearly
worse (digest-batch/s collapses 6->2). The default heuristic is well-tuned for this
GPU. No win. Consistent with the "Metal optimization chapter CLOSED" conclusion -
the upstream defaults (kernels, batch, concurrency) are all already at the practical
ceiling for this M31-integer-matmul workload on Apple GPUs. Methodology note: laptop
thermal drift makes single-sample cross-config comparison unreliable; ALWAYS use
short interleaved A/B (drift cancels) on warm-but-not-throttled state.

## CUDA seed-schedule hoist: transfers structurally, NEGLIGIBLE economically on Metal (2026-06-15, iter 3)

The CUDA path found a real ~6% win: hoist the seed-constant SHA message-schedule
words w[16..22] in the matrix-gen-from-seed kernel (they don't depend on the
per-element index, yet are recomputed for every matrix element). Checked whether
it transfers to Metal's `generate_base_matrix_from_seed` ->
`oracle_candidate_from_seed_and_index` (matmul_accel_kernels.metal:304/400).

STRUCTURE CONFIRMS: w[0..7]=seed (constant), w[8]=index (the only per-element
input word), w[9]=0x80000000, w[15]=288, w[10..14]=0 (all constant for retry=0,
which succeeds ~always since `candidate & MODULUS < MODULUS` fails at prob 2^-31).
The schedule recurrence w[i]=sigma1(w[i-2])+w[i-7]+sigma0(w[i-15])+w[i-16] makes
w[16..22] reference only constant words; w[23] is the FIRST to touch w[8] (via
sigma0(w[8])). So exactly w[16..22] are seed-constant and hoistable - identical to
CUDA. Recomputed for all N*N=262,144 elements/dispatch.

ECONOMICS KILL IT: the live Metal trace puts input/oracle gen at only ~6.9% of
GPU time (the batched matmul digest is 80.6%). Hoisting 7 of ~48 schedule words in
a 64-round SHA saves a fraction of that 6.9% -> <1% net, BELOW the +-13% thermal
noise floor on this laptop (so unmeasurable even if real). The CUDA win was ~6%
because matrix-gen IS the live hot kernel there; on Metal the matmul dominates and
matrix-gen is a rounding error. NOT pursued (no build/validate cycle spent).

Also checked `BTX_MATMUL_CPU_CONFIRM` (default ON, Metal+CUDA): CPU-reconfirms each
GPU candidate. Candidates are rare (prefilter survivors), so ~zero throughput cost,
and disabling trades away a safety net on an already-byte-exact backend. Not a lever.

## LOOP CONCLUSION (after 3 iterations): Metal pool optimization space is exhausted

Three self-paced /loop iterations, all rejected, converging with the prior
"optimization chapter CLOSED" finding:
1. solver-threads / pool-slots concurrency in pool mode -> FLAT (batch=192 nonce-
   prefilter path is throughput-bound; the bench 6x was a batch=1 strawman).
2. nonce-seed batch size sweep (192 default vs 256/384/512) -> 192 already optimal.
3. CUDA seed-schedule hoist -> transfers but <1% (matrix-gen is 6.9% of GPU here).

Root cause is structural: ~81% of Metal GPU time is the M31-integer batched matmul,
proven memory/occupancy-bound (every ALU/tiling/concurrency lever flat). The real
throughput wins live on the 5090/CUDA rig (e.g. the live ~6% w[16..22] hoist in
GenerateBaseMatrixFromSeedMidstateKernel). The only Metal upside left is a
structural redesign of the nonce-seeded prepare-overlap pipeline (days of
consensus-adjacent pow.cpp work, uncertain payoff, on a laptop ~400x slower than
the 5090) - not loop-sized. Stopping the self-paced loop; recommend redirecting
optimization effort to CUDA.
